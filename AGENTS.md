# gpt2api · Repo Workflow Notes

本文件用于约束本仓库内“只要改了文件”后的默认收尾流程。适用范围包括但不限于：编码、修 bug、改配置、改脚本、改文档、改前端、改部署、打包。

## 默认工作流

只要任务导致仓库文件发生变更，默认按下面顺序执行；除非用户明确要求跳过，或某一步与本次改动完全无关且已在最终说明里写清楚原因。

1. **完整测试**
   - 单元 / 静态检查：
     - `go vet ./...`
     - `go test ./...`
   - 前端构建校验（涉及 `web/`、静态资源、管理后台、页面交互时默认执行）：
     - `cd web && npm run build`
   - 端到端（e2e）冒烟：
     - 先启动服务（如 `docker compose -f deploy/docker-compose.yml up -d --wait`）
     - 再执行 `cd scripts && npm run smoke:docker`
   - 像素级 / 视觉回归测试：
     - 只要改动影响 UI、样式、截图、页面结构、渲染结果，就应执行像素级校验或等价的截图对比。
     - 如果仓库里没有现成的像素级测试，优先补一个最小可复现的视觉回归检查，或至少产出可核对的前后截图证据，不能直接略过不说明。

2. **提交 GitHub PR**
   - 完整测试通过后再提交 PR。
   - PR 描述中应包含：
     - 改动动机
     - 关键实现点
     - 测试结果
     - 如果有 UI 改动，附截图或视觉对比证据

3. **本地 Code Review**
   - PR 提交后，本地完成 Code Review。
   - 收到 review 意见后，先改代码，再重新从“完整测试”开始整轮执行。

4. **循环直到满足合并条件**
   - 只要因为 review、测试失败、联调问题而再次修改了文件，就重新执行：
     - 完整测试
     - 更新 PR
     - Code Review
     - 修改代码
   - 直到：
     - 单元测试通过
     - e2e 通过
     - 像素级/视觉校验通过
     - Code Review 通过
   - 满足以上条件后再合并 PR。

5. **打包代码**
   - 在完成代码修改并达到可交付状态后，默认补做打包，至少产出一个可运行文件、可部署产物或安装包。
   - 本仓库优先使用以下命令：
     - 本地二进制：`make build`
       - 产物：`bin/gpt2api`
     - 部署产物：`bash deploy/build-local.sh`
       - 产物：`deploy/bin/gpt2api`
       - 产物：`deploy/bin/goose`
       - 产物：`web/dist/`
   - 如果任务目标本身就是发布、交付、部署或安装包，则必须在最终说明里明确列出产物路径和验证方式。

## 结果汇报要求

完成任务时，最终说明应尽量包含：

- 改了哪些文件
- 执行了哪些测试，哪些通过，哪些因无关而跳过
- PR / Review 当前状态
- 打包产物路径
- 如果没有完成某一步，明确阻塞点和下一步

## 说明原则

- “最好按这个 workflow 执行”在本仓库里视为默认要求，而不是可无视建议。
- 若受限于当前环境无法直接发 PR、`@Copolot`、合并 PR 或执行某项测试，必须明确说明阻塞原因，不能假装该步骤已完成。
- 若改动仅限纯文档且确实不影响运行逻辑，可按实际情况跳过 e2e / 像素级校验，但要在最终说明中写明依据。
