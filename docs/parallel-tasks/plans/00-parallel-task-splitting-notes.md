# 04. 本次并行拆分任务说明

## 1. 文档目的

本文件用于解释这次为什么把改造任务拆成多份文档、多个阶段、多个 agent。

它不是新的需求文档,也不是新的实现计划,而是对以下三份核心文档的**使用说明**:

1. `/Users/swimmingliu/data/github-proj/gpt2api/docs/parallel-tasks/plans/01-image-only-admin-slimming-design.md`
2. `/Users/swimmingliu/data/github-proj/gpt2api/docs/parallel-tasks/plans/02-account-pool-runtime-gap-design.md`
3. `/Users/swimmingliu/data/github-proj/gpt2api/docs/parallel-tasks/plans/03-image-only-account-pool-refactor-plan.md`

## 2. 为什么要拆成三类文档

这次改造不是单一问题,而是两个不同层面的改造叠加:

### 2.1 产品边界收缩

核心问题是:

- 什么功能不再要
- 系统最终要长成什么样
- 哪些模块是历史包袱,哪些是必须保留

这部分由 `01` 文档负责。

### 2.2 账号池运行时补齐

核心问题是:

- 当前账号池到底缺什么
- 为什么现在还不算真正的账号池模式
- 后续运行时主链路应该怎么改

这部分由 `02` 文档负责。

### 2.3 工程实施拆分

核心问题是:

- 多个 agent 分别做什么
- 谁能并行,谁必须最后集成
- 每个任务应该改哪些文件、不要碰哪些文件

这部分由 `03` 文档负责。

如果把三类内容混在一份文档里,会出现两个问题:

1. agent 很难判断自己应该遵循“需求边界”还是“实现步骤”
2. 并行执行时,不同 agent 会反复修改同一份大文档,协作成本高

## 3. 本次拆分任务的核心判断

这次工程的本质不是“给当前项目加一个账号池功能”,而是:

1. 先把项目从 SaaS 网关收缩成图片账号池服务
2. 再把现有“只建模了池,但运行时没真正按池调度”的逻辑补齐

因此必须分成两个主轴:

- **主轴 A: 清理不需要的功能**
- **主轴 B: 补齐真正的账号池运行时**

如果不先拆开,实现时很容易出现下面的问题:

- 一边删用户 / 计费 / 充值,一边又让新调度逻辑继续依赖这些模块
- 前端已经改成纯管理后台,后端还保留大量用户侧接口
- 账号池表存在,但请求还是从全局账号列表调度

## 4. 为什么任务要按 6 个实现任务拆

`03` 文档里把工作分成了 6 个 Task,原因如下。

### Task 1: 边界冻结

这是为了先把目标定死。

如果没有这个步骤,后面的 agent 可能会各自做不同假设:

- 有人认为要保留 API Key
- 有人认为要删模型管理
- 有人认为图生图要保留

边界不冻结,后面的代码改造就会互相打架。

### Task 2: 后端 SaaS 能力剥离

这一块负责删后端历史包袱:

- 用户
- 计费
- 充值
- 统计
- 审计
- 备份
- 聊天

这是一个相对独立的后端改造面,适合单独给一个 agent。

### Task 3: 前端精简

前端和后端写集天然不同,可以独立推进。

把它单独拆出来有两个好处:

1. 不阻塞后端调度核心开发
2. 不让前端 agent 去碰 `main.go`、`router.go` 之类高冲突文件

### Task 4: 账号池运行时内核补齐

这是本次最关键的“新增核心能力”任务。

它的重点不是删代码,而是改运行时:

- 路由到池
- 池内候选集
- 成员过滤
- fallback

这部分需要专注调度和数据结构,不适合和大规模 SaaS 删除混在一个 agent 里做。

### Task 5: 网关与鉴权收口集成

这个任务之所以单独拆出来,是因为它天然会碰高冲突文件:

- `cmd/server/main.go`
- `internal/server/router.go`
- `internal/gateway/images.go`

如果多个 agent 同时改这些文件,冲突概率非常高。

因此它更适合放在前面几个任务完成后,由单独的集成 agent 收口。

### Task 6: 总集成与交付

最后必须留一个 Integrator 做总收尾:

- 清理残留依赖
- 更新 README
- 跑验证
- 决定数据库是“断依赖但暂不删表”还是“直接删表”

这个任务不应提前做,也不适合和并行开发同时做。

## 5. 推荐的 agent 分发方式

推荐直接按下面分:

- **Agent A**
  - 负责 `03` 文档中的 Task 2
  - 关键词: 后端 SaaS 剥离

- **Agent B**
  - 负责 `03` 文档中的 Task 3
  - 关键词: 前端精简为纯管理后台

- **Agent C**
  - 负责 `03` 文档中的 Task 4
  - 关键词: 账号池运行时补齐

- **Agent D**
  - 负责 `03` 文档中的 Task 5
  - 关键词: 网关与鉴权收口集成

- **Integrator**
  - 负责 Task 6
  - 关键词: 合并、清理、验证、交付

如果你还想再拆细,也可以继续分:

- Agent C1: 调度器按池筛选
- Agent C2: fallback 与成员参数
- Agent B1: 删除用户侧页面
- Agent B2: 补账号池后台页面

但再往下拆会明显增加集成成本,通常不建议第一轮就拆得太碎。

## 6. 任务之间的依赖关系

### 6.1 强依赖

- Task 5 依赖 Task 2 和 Task 4
- Task 6 依赖 Task 2 / 3 / 4 / 5

### 6.2 弱依赖

- Task 3 与 Task 2 基本无强依赖,可并行
- Task 4 与 Task 2 也可并行,只要不同时修改集成入口文件

### 6.3 为什么不让所有任务都并行

因为有些文件是天然收口点:

- `/Users/swimmingliu/data/github-proj/gpt2api/cmd/server/main.go`
- `/Users/swimmingliu/data/github-proj/gpt2api/internal/server/router.go`
- `/Users/swimmingliu/data/github-proj/gpt2api/internal/gateway/images.go`

这些文件一旦被多个 agent 同时修改,后面合并成本会非常高。

所以本次策略是:

- **能并行的并行**
- **高冲突文件最后集中收口**

## 7. 你给 agent 下发任务时建议附带的信息

建议你给每个 agent 至少附带:

1. 它负责的文档编号
2. 它只允许修改的目录 / 文件
3. 它不能碰的高冲突文件
4. 它完成后必须执行的验证命令
5. 它的产出要满足的验收点

例如:

- “你负责 `03` 文档的 Task 4,只改 `internal/accountpool/**`、`internal/scheduler/**`、`internal/account/importcore/**`,不要动 `main.go` 和 `router.go`。”

这样能显著减少 agent 之间的交叉覆盖。

## 8. 本次拆分的最终目标

本次拆分不是为了把任务写得好看,而是为了让后续并行开发满足三个结果:

1. **边界清晰**
   - 先删掉不需要的产品能力

2. **实现聚焦**
   - 账号池运行时单独补齐,不被 SaaS 逻辑拖累

3. **并行可控**
   - 多个 agent 可以同时推进,但不会在收口文件上大量冲突

## 9. 最简使用顺序

如果你后面要直接发给 agent,建议按这个阅读顺序给他们:

1. 先看 `01`
2. 再看 `02`
3. 最后按 `03` 领任务
4. 如需理解为什么这么拆,再看本文件 `04`

一句话总结:

`01` 负责定边界,`02` 负责定能力缺口,`03` 负责定实施顺序,`04` 负责解释为什么这样拆。
