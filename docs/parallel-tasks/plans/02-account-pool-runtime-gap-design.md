# 02. gpt2api 账号池运行时缺口与目标设计文档

**日期**: 2026-04-26

## 1. 文档定位

本文档回答两个问题:

1. 当前仓库虽然已经有账号池表、成员表、模型路由表,为什么运行时还不算真正的账号池模式
2. 在“仅保留图片网关 + 管理后台”的目标下,账号池运行时 MVP 应该冻结成什么样

这不是新的产品边界文档,而是对“账号池运行时主链路”做收口设计。后续并行 agent 应优先把本文当作:

- 运行时缺口清单
- Phase 1 共享契约
- Task 4 / Task 5 的接口依据

## 2. 共享契约摘要

以下约定从本文起冻结,后续实现与 review 默认以此为准。

### 2.1 系统边界

- 对外网关只保留:
  - `POST /v1/images/generations`
  - `GET /v1/models`
  - `GET /p/img/:task_id/:idx`
- 运行时只围绕 `gpt-image-2` 图片生成展开
- 后台只保留管理员视角,不再依赖普通用户、计费、充值、用户 API Key
- 网关鉴权收敛为实例级静态 Bearer Token 或等价的最小系统级 token

### 2.2 账号池运行时总原则

- 图片请求进入调度前,必须先得到明确的账号池路由
- 图片主链路不得再把“全局账号列表调度”当成主要逻辑
- 主池优先,只有主池完全无可用成员时才允许切 fallback
- 池成员语义优先级高于“旧全局兼容逻辑”;兼容模式最多只作为迁移期临时兜底,不是 image-only MVP 的常态

### 2.3 给后续 agent 的一句话契约

`gpt-image-2` 请求必须走“默认主池或显式路由池 -> 池成员候选集 -> 主池优先调度 -> fallback pool -> 图片 runner”,而不是“全局账号表扫一遍再顺手带上池字段”。

## 3. 背景

当前仓库已经具备一部分账号池基础设施:

- `account_pools`
- `account_pool_members`
- `model_pool_routes`
- 账号池 CRUD API
- 导入账号时指定 `target_pool_id`

但当前调度主链路仍然是“全局账号列表调度”,账号池更像是:

- 数据建模已存在
- 后台管理接口已存在
- 导入后归类能力已存在
- 运行时调度尚未真正切到“池内治理”

因此,当前系统还不是可用的 `sub2api` 风格图片账号池服务。

## 4. 当前已实现 vs 关键缺口

### 4.1 已实现

1. **账号池数据建模**
   - 池
   - 池成员
   - 模型到池路由

2. **账号池管理 API**
   - 池 CRUD
   - 成员增删改查
   - 模型路由 CRUD

3. **导入后入池**
   - 统一导入链路已支持 `target_pool_id`

### 4.2 关键缺口

1. **请求并未按池调度**
   - 调度器当前仍扫全局可调度账号
   - 图片请求没有先解析“目标池”

2. **模型路由未接入图片运行时**
   - `model_pool_routes` 还没有决定图片请求的候选集边界

3. **成员参数未真正进入调度**
   - `enabled`
   - `priority`
   - `weight`
   - `max_parallel`

4. **fallback 只有表字段,没有真实回退链路**
   - `fallback_pool_id` 未成为主池失败后的运行时兜底

5. **sticky 只有扩展位,没有稳定接口**
   - `sticky_ttl_sec` 还未进入请求级路由

6. **池级可观测性不足**
   - 后台无法清晰看到池的可用成员、繁忙成员、fallback 命中等最小治理信息

## 5. 总目标

在精简后的“图片账号池服务”里,把现有账号池补全成真正可用的运行时能力,让 `gpt-image-2` 请求可以:

1. 命中明确的主池
2. 只在该池成员内调度
3. 在主池耗尽时按规则切到 fallback pool
4. 让成员级参数进入真实决策
5. 为 sticky 和池级并发治理留出稳定扩展点
6. 让后台至少能看清路由关系和运行时命中情况

### 5.1 范围限定

本轮只围绕**图片生成**展开,不考虑:

- chat 模型池
- codex 模型池
- 多平台资源池
- 多租户套餐路由
- 普通用户 API Key 到池的动态绑定

## 6. 目标运行时模型

### 6.1 请求主链路

Phase 1 目标图片请求流程:

1. 请求进入 `POST /v1/images/generations`
2. 网关识别逻辑模型为 `gpt-image-2`
3. 网关先解析本次请求的 `DispatchRoute`
4. 调度器只从主池成员中构造候选集
5. 若主池无可用成员,且允许 fallback,再切换 fallback pool
6. 选中账号后继续走现有图片 runner

### 6.2 MVP 路由策略

MVP 先收敛到最小策略:

- 所有图片请求默认走一个配置好的主池,例如 `image-main`
- 若启用 fallback,再切到 `image-fallback`
- 不要求本轮就支持“按 header 指定不同池”

这意味着 Phase 1 先解决“池真的进入运行时”,而不是一开始就做复杂的多来源动态路由。

### 6.3 后续扩展位

MVP 稳定后再扩展:

- 按 header 指定池
- 按路径或来源映射池
- 按系统级 token 绑定池
- 按业务来源做 sticky

## 7. 运行时共享契约

### 7.1 DispatchRoute

图片网关在进入调度前必须解析出一个稳定的 `DispatchRoute`。MVP 至少需要:

- `primary_pool_id`
- `fallback_pool_id`
- `allow_fallback`
- `reason`
  - 例如 `default-image-pool`
  - 或 `model-route`

建议约束:

- `DispatchRoute` 是“这次请求应该去哪组池”的网关层契约
- 它不负责任何账号健康判断
- 没有 `DispatchRoute` 的图片请求不应直接回到“全局账号列表调度”

### 7.2 ResolvedPool

调度器在拿到 `DispatchRoute` 后,应把每次池尝试表达成 `ResolvedPool` 语义。MVP 至少要能表达:

- 当前尝试的池 ID
- 当前是否 fallback 命中
- 当前池的成员候选集
- 当前池失败的原因

推荐用途:

- 记录日志
- 返回调度失败原因
- 后台后续接入最小可观测性

### 7.3 image-only 迁移约束

历史 `LegacyGlobal` 兼容逻辑可暂时保留在底层服务或过渡层,但图片主链路必须把它视为:

- 迁移阶段临时兜底
- 不能作为 Phase 1 验收通过的依据

只要 image-only MVP 已落地,图片请求就不应再依赖全局账号列表作为主路径。

## 8. 调度器缺口与目标

### 8.1 当前缺口

- 当前调度入口更像 `Dispatch(ctx, modelType string)`
- 候选账号来自全局账号表
- 账号池成员字段没有进入真实过滤与排序

### 8.2 目标要求

调度器输入应从“模型类型”升级为“调度路由 + 请求上下文”:

- 候选账号必须来自池成员查询,不是全局列表
- 成员筛选至少要考虑:
  - 成员 `enabled`
  - 池是否启用
  - 账号健康状态
  - cooldown
  - 最小间隔
  - 日额度
  - 最近使用时间

### 8.3 主池 / fallback 行为

- 先完整尝试主池
- 只有主池完全无可用成员时,才允许进入 fallback
- fallback 命中必须可观测:
  - 日志可见
  - 后台后续可统计
- 主池仍有可用成员时,不得抢先使用 fallback

## 9. 池成员参数语义冻结

### 9.1 `enabled`

- 成员级开关
- 关闭后,该账号在**当前池**不可调度
- 不影响它在其他池中的可用性

### 9.2 `priority`

Phase 1 在文档中冻结为:

- **数值越小,优先级越高**
- 默认值 `100` 表示普通优先级
- 调度时先按 `priority ASC` 分层,再做下一步选择

这样做的原因:

- 和现有默认值 `100` 兼容
- 运维可以用更小的值给“保底高质量账号”提权
- 不需要引入额外枚举

### 9.3 `weight`

Phase 1 语义冻结为:

- 只在**相同 `priority` 层内**使用
- 数值越大,理论上被选中的权重越高
- 本轮可以先不做复杂随机权重算法
- 但字段不能保持“存在却完全无语义”

MVP 推荐退化实现:

- 同优先级内先按健康 / 冷却 / 最近使用时间筛掉不可选项
- 若仍有多个候选,可先做简单加权随机,或保留一个稳定、可解释的近似策略
- 无论选哪种策略,都必须在注释或测试里写清楚

### 9.4 `max_parallel`

Phase 1 语义冻结为:

- 表达“该账号在该池内允许承接的并发上限”
- 若当前全局运行时仍坚持“一号一任务”,则本轮可退化为 `1`
- 但调度代码必须至少读取并解释这个字段,不能完全忽略

### 9.5 `sticky_ttl_sec`

- Phase 1 可以不实现真实 sticky
- 但接口与数据结构要留出稳定扩展点
- 后续 sticky key 候选包括:
  - 系统级 token
  - 请求来源标识
  - 自定义业务 key

## 10. 网关层缺口与目标

### 10.1 当前缺口

- 图片网关没有显式“目标池解析”步骤
- `/v1/models`、模型表、池路由之间的关系没有进入主链路

### 10.2 目标要求

- 图片请求在进入调度前必须先得到 `DispatchRoute`
- `GET /v1/models` 在 image-only 目标下可以只暴露 `gpt-image-2`
- 若存在模型路由表,优先把模型到池的关系固化在后台
- 若模型路由未命中,可回落到“默认图片主池配置”,但不能回落到“全局账号列表调度”

## 11. 管理后台与可观测性目标

当前后台至少需要能从“池视角”治理系统,而不是只从“账号列表视角”看系统。

MVP 需要保留或补齐:

- `/api/admin/account-pools`
- `/api/admin/account-pools/:id/members`
- `/api/admin/account-pool-routes`
- `/api/admin/accounts`
- `/api/admin/proxies`

MVP 最小可见信息:

- 池列表
- 池成员关系
- 模型到池路由
- 请求日志里的主池 / fallback 命中信息

后续再补:

- 池级可用成员数
- 池级繁忙数
- fallback 命中率
- 池级熔断 / 限流状态

## 12. 分阶段实现建议

### Phase 1: 让池真正参与图片请求调度

必须完成:

1. 图片请求解析主池
2. 调度器按池成员筛选账号
3. 成员 `enabled` 生效
4. 主池无可用成员时切 fallback
5. 后台能看见模型路由与池成员关系

这是本轮最关键的交付。没有它,就还不是账号池运行时。

### Phase 2: 完善池内调度策略

补齐:

- `priority`
- `weight`
- `max_parallel`
- fallback 统计
- 基础池级看板

### Phase 3: 增强治理能力

再考虑:

- sticky 路由
- 池级限流
- 池级熔断
- 更丰富的按来源到池的映射策略

## 13. 配置项

池运行时至少需要:

- 默认图片主池编码或 ID
- 是否启用 fallback
- fallback 池编码或 ID
- 是否启用 sticky
- sticky TTL 默认值

推荐优先级:

1. 先用配置驱动默认主池
2. 再用后台模型路由覆盖默认值
3. 最后再做更细粒度的请求级路由

## 14. 验收标准

满足以下条件时,可判定“账号池运行时 MVP 已落地”:

1. 图片请求进入后,日志能明确看到命中的目标池
2. 调度器只从目标池成员中选账号
3. 禁用某个成员后,该账号在该池不再被选中
4. 主池无可用账号时,请求可切到 fallback pool
5. 后台能查看:
   - 池列表
   - 池成员
   - 模型到池路由
6. 导入账号时指定 `target_pool_id` 后,账号能自动成为该池成员
7. 图片运行时不再依赖“全局账号列表调度”作为主要逻辑

## 15. 非目标提醒

本文解决的是:

- 当前账号池运行时还缺什么
- image-only 目标下应如何冻结账号池主链路

本文不解决:

- 是否保留 SaaS 功能
- 是否立刻删库删表
- chat/codex 的池化策略
- 多租户套餐路由

换句话说:

- 产品边界收缩是另一份文档的问题
- 本文只负责把“账号池真正进入运行时”说清楚
