# 02. gpt2api 账号池运行时缺口与目标设计文档

**日期**: 2026-04-26

## 1. 背景

当前仓库已经具备一部分“账号池基础设施”,但离真正可用的 `sub2api` 风格账号池运行时还有明显距离。

已存在的基础能力包括:

- 账号池表、池成员表、模型到池路由表
- 账号池 CRUD API
- 导入账号时指定 `target_pool_id`
- 账号、代理、刷新、额度探测基础链路

但当前调度主链路仍然是“全局账号列表调度”,并没有真正进入“按池路由、按池筛选、按池治理”的运行时模式。

## 2. 当前已实现 vs 未实现

### 2.1 已实现

1. **账号池数据建模**
   - `account_pools`
   - `account_pool_members`
   - `model_pool_routes`

2. **账号池管理 API**
   - 池 CRUD
   - 成员增删改查
   - 模型路由 CRUD

3. **导入后入池**
   - 统一导入链路已经支持 `TargetPoolID`

### 2.2 关键未实现

1. **请求并未按池调度**
   - 调度器当前仍扫全局可调度账号
   - 请求模型没有真正参与池路由解析

2. **模型路由未接入运行时**
   - `model_pool_routes` 存在,但网关没有基于它决定调度范围

3. **池成员参数未生效**
   - `weight`
   - `priority`
   - `max_parallel`
   - `enabled`
   这些参数还没有完整进入调度决策

4. **池级 fallback 未生效**
   - `fallback_pool_id` 还没有成为实际调度兜底逻辑

5. **sticky_ttl_sec 未生效**
   - 池定义里已有粘性 TTL 字段,但还没有请求级粘性路由实现

6. **池级并发治理缺失**
   - 当前只有“单账号锁”
   - 还没有“池内成员并行度”和“池整体行为”的治理能力

7. **池级可观测性不足**
   - 后台还不能清晰看到每个池的可用账号数、繁忙数、熔断数、命中率

## 3. 目标

### 3.0 MVP 验收口径（冻结）

- 请求进入 `POST /v1/images/generations` 后,必须先解析出稳定的 `DispatchRoute`
- 请求必须能命中**主池**
- 主池成员必须真实参与调度,`enabled` 成员开关必须生效
- 主池无可用成员时,必须可切入 `fallback_pool_id`
- 调度器不得再依赖“全局账号列表”作为图片主路径候选集
- 导入时写入 `TargetPoolID` 的账号,必须可被新调度器直接看到

### 3.1 总目标

在精简后的“图片账号池服务”里,把现有账号池补全成真正可用的运行时能力,让 `gpt-image-2` 请求可以:

1. 命中指定账号池
2. 只在该池成员内调度
3. 支持主池 / 兜底池
4. 支持成员级调度参数
5. 支持池级粘性与并发治理
6. 支持后台管理与最小可观测性

### 3.2 范围限定

本轮设计只围绕**图片生成**展开,不再考虑:

- chat 模型池
- codex 模型池
- 多平台资源池
- 多模型 SaaS 套餐路由

## 4. 目标运行时模型

### 4.1 请求路由流程

精简后建议的图片请求流程:

1. 请求进入 `POST /v1/images/generations`
2. 网关识别当前逻辑模型固定为 `gpt-image-2`
3. 网关解析该请求目标池
4. 调度器仅在该池成员内选择账号
5. 若主池无可用成员,按规则切到 fallback pool
6. 选中账号后继续走现有图片 runner

### 4.2 MVP 路由策略

MVP 建议先收敛到最简单的池路由规则:

- 所有图片请求默认走一个配置好的主池,例如 `image-main`
- 若启用了 fallback,则可切到 `image-fallback`

即:

- Phase 1 先解决“池真的生效”
- Phase 2 再扩展“按请求选择不同池”

### 4.3 后续扩展位

在 MVP 稳定后,再考虑:

- 按 header 指定池
- 按路径映射池
- 按系统级 API key 绑定池
- 按业务来源做 sticky

## 5. 运行时缺口明细

### 5.1 网关层缺口

当前缺口:

- 图片网关没有显式“目标池解析”步骤
- `/v1/models`、模型表、池路由三者关系不清晰

目标要求:

- 图片请求在进入调度前必须先得到一个明确的 `DispatchRoute`
- `DispatchRoute` 至少包含:
  - 主池 ID
  - 兜底池 ID
  - 是否允许回退

### 5.2 调度器缺口

当前缺口:

- `Dispatch(ctx, modelType string)` 只表达“模型类型”
- 实际候选集来自全局账号表

目标要求:

- 调度器输入从“模型类型”升级为“调度路由 + 请求上下文”
- 候选账号必须来自池成员查询,不是全局查询
- 调度排序至少支持:
  - 成员启用状态
  - 成员优先级
  - 账号健康状态
  - cooldown
  - 最小间隔
  - 日额度
  - 最近使用时间

### 5.3 池成员语义缺口

当前成员字段存在,但语义尚未完全落地。

目标语义:

- `enabled`
  - 成员级开关
  - 关闭后账号在该池不可调度,但不影响账号在其他池继续使用

- `priority`
  - 先用于粗粒度排序
  - 值越高越优先或值越低越优先,必须统一并文档化

- `weight`
  - 在相同优先级内做加权选择
  - Phase 1 可以先不实现随机权重,但字段语义要冻结

- `max_parallel`
  - 表达成员在该池内允许承接的并发
  - 若账号全局仍坚持“一号一任务”,则该字段 Phase 1 可先退化为 `1`
  - 但不能保持“字段存在、运行时完全无语义”

### 5.4 Fallback 缺口

当前 `fallback_pool_id` 已建模,但未进入真实调度路径。

目标要求:

- 主池完全不可用时,才切换 fallback
- fallback 命中必须在日志与后台上可见
- 不能在主池仍有可用账号时抢先使用 fallback

### 5.5 Sticky 缺口

当前池定义有 `sticky_ttl_sec`,但没有真实粘性路由。

目标要求:

- MVP 可以先不实现
- 但应在接口与数据结构上留出稳定扩展点

推荐粘性 key 候选:

- 服务级 API key
- 请求来源标识
- 自定义业务 key

### 5.6 管理后台缺口

当前后台还缺:

- 独立账号池管理视图
- 模型到池路由视图
- 池级状态汇总
- 账号所属池清晰展示

目标要求:

- 管理员能从“池视角”治理账号,而不是只从“账号列表视角”看系统

## 6. 推荐分阶段实现

### Phase 1: 让池真正参与请求调度

本阶段必须完成:

1. 图片请求解析主池
2. 调度器按池成员筛选账号
3. 成员 `enabled` 生效
4. 主池无可用成员时切 fallback
5. 后台能看见模型路由和池成员关系

这是本轮最关键的交付,没有它就还不是账号池运行时。

### Phase 2: 完善池内调度策略

本阶段补齐:

- 成员 `priority`
- 成员 `weight`
- 成员 `max_parallel`
- fallback 统计
- 基础池级看板

### Phase 3: 增强治理能力

本阶段再考虑:

- sticky 路由
- 池级限流
- 池级熔断
- 请求来源到池的路由策略扩展

## 7. 精简版系统下的目标接口

### 7.1 后台管理接口

应保留或完善:

- `/api/admin/account-pools`
- `/api/admin/account-pools/:id/members`
- `/api/admin/account-pool-routes`
- `/api/admin/accounts`
- `/api/admin/proxies`

### 7.2 对外网关接口

只围绕图片:

- `POST /v1/images/generations`
- `GET /v1/models`
- `GET /p/img/:task_id/:idx`

### 7.3 配置项

池运行时至少需要:

- 默认图片主池编码或 ID
- 是否启用 fallback
- fallback 池编码或 ID
- 是否启用 sticky
- sticky TTL 默认值

## 8. 验收标准

满足以下条件时,可判定“账号池运行时 MVP 已落地”:

1. 图片请求进入后,后台日志能明确看到命中的目标池
2. 调度器只从目标池成员中选账号
3. 禁用某个成员后,该账号在该池不再被选中
4. 主池无可用账号时,请求可切到 fallback pool
5. 后台能查看:
   - 池列表
   - 池成员
   - 模型到池路由
6. 导入账号时指定 `target_pool_id` 后,账号能自动成为该池成员
7. 运行时不再依赖“全局账号列表调度”作为主要逻辑

## 9. 与功能精简文档的关系

这份文档假设系统已经完成如下产品收缩:

- 无文本聊天
- 无计费充值
- 无用户自助体系
- 只保留管理员后台
- 只保留图片生成

也就是说:

- 本文解决的是**账号池还差什么**
- 不是讨论“还要不要保留 SaaS 功能”

前者是运行时能力问题,后者是产品边界问题,两者需要分开实施。
